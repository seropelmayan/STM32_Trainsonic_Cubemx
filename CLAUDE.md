# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`Ropetow` is STM32 firmware for sensored **Field-Oriented Control (FOC)** of a single PMSM, generated by **ST Motor Control Workbench (X-CUBE-MCSDK v6.4.2)** on top of STM32CubeMX (6.17.0). Target MCU: **STM32G431CBTx** (Cortex-M4F, FPU `fpv4-sp-d16` hard-float, 128 KB flash / 32 KB RAM).

The motor profile (see [Inc/pmsm_motor_parameters.h](Inc/pmsm_motor_parameters.h)) is a high-pole-count (40 pole-pairs), low-speed (700 rpm max), high-current (~29 A) direct-drive PMSM with a 4096-PPR quadrature encoder — consistent with a rope-tow / winch drive. Speed/position feedback is the **TIM3 encoder**; current sensing uses **ICS (isolated current sensors)** via `ics_g4xx_pwm_curr_fdbk`.

## Critical: this is a generated project — do not hand-edit generated regions

The vast majority of `Src/` and `Inc/` is regenerated by the Motor Control Workbench / CubeMX from [Ropetow.ioc](Ropetow.ioc) (CubeMX peripheral config) and [Ropetow.ioc.wb](Ropetow.ioc.wb) (MC Workbench motor/drive config). Regeneration **overwrites** these files.

- **Persistent hand-written code must live only between `/* USER CODE BEGIN x */` and `/* USER CODE END x */` markers** (and `USER SECTION BEGIN/END` in hook files). Anything outside these guards is lost on the next generation.
- The intended extension point for application logic is [Src/mc_app_hooks.c](Src/mc_app_hooks.c) — `MC_APP_BootHook()` (end of MC boot) and `MC_APP_PostMediumFrequencyHook_M1()` (after each medium-frequency task). Put motor start/stop/ramp orchestration here, inside the USER SECTION guards.
- To change motor parameters, pinout, clocks, current-sensing topology, or control gains, **edit the `.ioc`/`.wb` in the tools and regenerate** rather than editing the generated headers (`drive_parameters.h`, `parameters_conversion*.h`, `mc_config*.c`, etc.) by hand.
- `Ropetow.ioc.bak` is a backup; `Ropetow.log` is a generation log — not source.

## Architecture (layers, outer → inner)

1. **Application / control entry — [Src/main.c](Src/main.c)**: standard CubeMX `main()`. Brings up clocks (170 MHz from 24 MHz HSE), GPIO, ADC1, CORDIC, TIM1 (PWM), TIM3 (encoder), then `MX_MotorControl_Init()` and NVIC. The `while(1)` loop is empty — all real-time work runs in interrupts.
2. **Public motor API — [Inc/mc_api.h](Inc/mc_api.h) / [Src/mc_api.c](Src/mc_api.c)**: the surface you call to command the motor. Key calls: `MC_StartMotor1()`, `MC_StopMotor1()`, `MC_ProgramSpeedRampMotor1[_F]()`, `MC_ProgramTorqueRampMotor1[_F]()`, `MC_SetCurrentReferenceMotor1()`, `MC_GetSTMStateMotor1()`, `MC_AcknowledgeFaultMotor1()`, `MC_GetOccurredFaultsMotor1()`. There is one motor ("Motor1") only.
3. **Scheduler / state machine — [Src/mc_tasks.c](Src/mc_tasks.c) + [Src/mc_tasks_foc.c](Src/mc_tasks_foc.c)**: `MC_RunMotorControlTasks()` drives the medium-frequency task (speed loop, state machine); the high-frequency FOC current loop runs from the ADC/TIM ISR. The motor state machine is reached through [Src/mc_interface.c](Src/mc_interface.c) (`MCI_*`).
4. **Drivers / MC library — `MCSDK_v6.4.2-Full/MotorControl/MCSDK/MCLib`**: the unmodified MCSDK component library (PID, ramp manager, encoder feedback, circle limitation, bus-voltage/temperature sensors, ICS current feedback). Treat as vendor code; don't edit.
5. **HAL / CMSIS — `Drivers/`**: STM32G4xx HAL and CMSIS. Vendor code.

### Interrupt model (timing-critical — see [Src/main.c](Src/main.c) `MX_NVIC_Init`)
- `TIM1_UP_TIM16_IRQn` — priority **0** (highest): FOC high-frequency PWM update.
- `ADC1_2_IRQn` — priority 2: current-reading completion feeding the FOC loop.
- `TIM3_IRQn` — priority 3: encoder.
- `TIM1_BRK_TIM15_IRQn` — priority 4: PWM break / overcurrent protection.
ISR bodies live in [Src/stm32g4xx_mc_it.c](Src/stm32g4xx_mc_it.c) and [Src/stm32_mc_common_it.c](Src/stm32_mc_common_it.c). When adding work, respect this priority ordering — never block the high-frequency loop.

### Parameter flow
`pmsm_motor_parameters.h` + `drive_parameters.h` + `power_stage_parameters.h` (physical units) → `parameters_conversion.h` / `parameters_conversion_g4xx.h` (convert to fixed-point/int16 and timer counts, e.g. `PWM_PERIOD_CYCLES`, `DEAD_TIME_COUNTS`, `M1_PULSE_NBR`) → consumed by `main.c` peripheral init and the MC library. Changing a physical parameter without regenerating leaves the converted constants stale.

## Build / flash / debug

There is **no command-line build in this tree** — building is done through **STM32CubeIDE** (the `STM32CubeIDE/` folder is the Eclipse project: [.project](STM32CubeIDE/.project), [.cproject](STM32CubeIDE/.cproject), linker script [STM32CubeIDE/STM32G431CBTX_FLASH.ld](STM32CubeIDE/STM32G431CBTX_FLASH.ld)).

- **Build**: open the project in STM32CubeIDE → build `Debug` or `Release` configuration (GNU arm-none-eabi toolchain; produces `Ropetow.elf`). Headless builds are possible via `stm32cubeidec --launcher.suppressErrors -nosplash -application org.eclipse.cdt.managedbuilder.core.headlessbuild -build Ropetow/Debug` but the IDE GUI is the normal path.
- **Flash / debug**: ST-LINK via STM32CubeIDE or `STM32_Programmer_CLI`. Monitor/tune live via the MC Workbench **Motor Pilot** over the configured comms link.
- **Regenerate code**: open `Ropetow.ioc` in MC Workbench/CubeMX and generate. The `ftl/` directory holds the FreeMarker templates the Workbench uses to emit the MC source.

Compile defines (from `.mxproject`): `ARM_MATH_CM4`, `USE_HAL_DRIVER`, `STM32G431xx`. Include roots: `Inc/`, the HAL/CMSIS `Drivers/` Inc dirs, and `MCSDK_v6.4.2-Full/.../MCLib/{Any,G4xx}/Inc`.

There are no unit tests, linters, or package scripts in this repository.

## Conventions

- C, embedded, fixed-point-heavy. Speeds/currents/voltages appear both as `int16_t` (internal MC scaling) and `_F` float variants in the API — match the convention of the function you're calling.
- Keep all edits inside USER CODE guards (see the "generated project" section above) or they will not survive regeneration.
